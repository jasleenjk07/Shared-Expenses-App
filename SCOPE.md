# SCOPE.md — Anomaly Log & Database Schema

## Database Schema

See [`database/schema.sql`](database/schema.sql) for the full DDL. Entity relationships:

```
users ──┬── group_memberships ── groups
        ├── expenses (paid_by)
        ├── expense_splits
        └── settlements (from/to)

import_sessions ── import_anomalies
groups ── expenses, settlements, import_sessions
```

**Key design choices:**
- All balances computed in **INR** (`amount_inr` on expenses); original currency preserved
- `group_memberships.joined_at` / `left_at` gate who participates in expense splits by date
- Settlements are separate from expenses (payments between members)
- Import anomalies stored for Meera's approval workflow

## CSV Anomalies Detected & Handling Policies

Each row references `Expenses_Export.csv` (1-indexed data rows; row 1 = header).

| # | Row | Anomaly Type | Detection | Policy / Action |
|---|-----|--------------|-----------|-----------------|
| 1 | 6 | `DUPLICATE_EXPENSE` | Same date, payer, amount, similar description ("Dinner at Marina Bites" / "dinner - marina bites") | Skip duplicate; keep first entry (row 5). Flag for Meera's approval. |
| 2 | 7 | `COMMA_IN_AMOUNT` | Amount `"1,200"` contains thousands separator | Strip commas before parsing → ₹1,200 |
| 3 | 9 | `PAYER_NAME_VARIANT` | Payer `priya` (lowercase) | Normalize to canonical `Priya` via alias map |
| 4 | 10 | `AMOUNT_ROUNDING` | Amount `899.995` exceeds 2 decimal places | Round half-up to ₹900.00 |
| 5 | 11 | `PAYER_NAME_VARIANT` | Payer `Priya S` | Map alias → `Priya` |
| 6 | 13 | `MISSING_PAYER` | Empty `paid_by` for house cleaning supplies | Import expense with null payer; exclude from balance until manually assigned. Requires approval. |
| 7 | 14 | `SETTLEMENT_AS_EXPENSE` | "Rohan paid Aisha back" with settlement keywords | Convert to `settlements` record: Rohan → Aisha, ₹5,000 |
| 8 | 15 | `PERCENTAGE_NOT_100` | Percentages 30+30+30+20 = 110% | Normalize proportionally (each × 100/110) before allocating |
| 9 | 20–21, 23, 26 | `FOREIGN_CURRENCY` | USD amounts treated as INR in spreadsheet | Convert at configurable rate (default ₹83/USD); store both original and `amount_inr` |
| 10 | 22 | `SHARE_SPLIT` | split_type `share` with share counts | Use weighted share allocation (not equal) |
| 11 | 23 | `UNKNOWN_SPLIT_MEMBER` | `Dev's friend Kabir` in split_with | Exclude from split; redistribute among known flatmates |
| 12 | 25 | `DUPLICATE_CONFLICT` | "Thalassa dinner" logged by Rohan (₹2,450) conflicts with Aisha's (₹2,400) | Keep first entry (row 24); skip conflicting row. Requires approval. |
| 13 | 26 | `NEGATIVE_AMOUNT` | Parasailing refund `-30 USD` | Treat as refund: negative `amount_inr` reduces payer's net contribution |
| 14 | 27 | `DATE_FORMAT_INCONSISTENT` | Date `Mar-14` | Parse as 14-Mar-2026 (year from context) |
| 15 | 27 | `PAYER_NAME_VARIANT` | Payer `rohan ` (trailing space) | Trim and normalize → `Rohan` |
| 16 | 28 | `MISSING_CURRENCY` | Empty currency field | Default to INR with warning |
| 17 | 31 | `ZERO_AMOUNT` | Swiggy dinner amount `0` | Skip row entirely — placeholder/fix note in CSV |
| 18 | 34 | `AMBIGUOUS_DATE` | `04-05-2026` could be 4-May or 5-Apr | Use DD-MM-YYYY → **2026-05-04** (May 4). Notes in CSV ask "April 5 or May 4?" — flagged for review. |
| 19 | 36 | `INACTIVE_MEMBER_IN_SPLIT` | Meera in April groceries split but left end-March | Remove Meera; redistribute share among active members (Sam's concern) |
| 20 | 42 | `SPLIT_TYPE_MISMATCH` | split_type `equal` but split_details has share values | Honor split_details; apply share-based split |

## Import Report

The app generates a JSON import report on every CSV upload, persisted in `import_sessions.report_json` and shown in the UI Import tab. Each anomaly includes: row number, type, description, action taken, and whether approval is required.

Sample report location after import: `reports/import_report.json` (generated by test script).
